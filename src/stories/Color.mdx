## Introduction

Application of the color palette brings unified and recognizable consistency to the portfolio. This consistency is grounded in a set of well-defined rules about how to work with role-based color tokens in the context of light and dark themes.

---

## Implementing color

The portfolio uses tokens and themes to manage color. Tokens are role-based, and themes specify the color values that serve those roles.

<table>
  <thead>
    <tr><th>Term</th><th>Definition</th></tr>
  </thead>
  <tbody>
    <tr>
      <td><strong>Theme</strong></td>
      <td>A collection of color values designed to create a specific look. Themes control the value assigned to every token. The portfolio has two themes: Light and Dark.</td>
    </tr>
    <tr>
      <td><strong>Token</strong></td>
      <td>A role-based identifier that assigns a color. Unlike hex codes, tokens apply universally across themes. For example, <code>--fg</code>, <code>--accent</code>, <code>--border</code>.</td>
    </tr>
    <tr>
      <td><strong>Role</strong></td>
      <td>The systematic usage of a color assigned to a token. Roles do not change between themes — <code>--fg</code> is always foreground text regardless of the active theme.</td>
    </tr>
    <tr>
      <td><strong>Value</strong></td>
      <td>The unique visual attribute (hex code or rgba value) assigned to a token through a theme. For example, <code>--fg</code> resolves to <code>#222222</code> in light and <code>#fffbf5</code> in dark.</td>
    </tr>
  </tbody>
</table>

## Themes

Themes serve as an organizational framework for color in the portfolio. Each theme is based on a primary background color. The portfolio has one light theme and one dark theme.

<table>
  <thead>
    <tr><th>Theme</th><th>Type</th><th>Primary background</th><th>Token</th><th>Value</th></tr>
  </thead>
  <tbody>
    <tr><td><strong>Light</strong></td><td>Light</td><td>Cream</td><td><code>--bg</code></td><td>#fffbf5</td></tr>
    <tr><td><strong>Dark</strong></td><td>Dark</td><td>Near-black</td><td><code>--bg</code></td><td>#1c1a16</td></tr>
  </tbody>
</table>

The active theme is controlled by the `.dark` class on `<html>`. When `.dark` is present, all adaptive tokens resolve to their dark values simultaneously. The `ThemeToggle` component manages this — no component-level logic is needed.

In the light theme, the surface layer (`--surface`) sits one step above the page background, creating depth. In the dark theme, the same layering applies in reverse — surface is one step lighter than the background.

## Accessibility

Using contrast is the most important consideration when making color choices. Awareness of WCAG standards is key to accessible color selection.

Contrast is the difference in brightness between any two elements. The Web Content Accessibility Guidelines (WCAG) set specific ratios that achieve the minimum required contrast for legibility.

<table>
  <thead>
    <tr><th>Element type</th><th>Minimum — AA</th><th>Enhanced — AAA</th></tr>
  </thead>
  <tbody>
    <tr><td>Body text (below 24px)</td><td>4.5:1</td><td>7:1</td></tr>
    <tr><td>Large text (24px and above)</td><td>3:1</td><td>4.5:1</td></tr>
    <tr><td>Non-text elements and icons</td><td>3:1</td><td>—</td></tr>
  </tbody>
</table>

All text-on-background combinations in the portfolio meet WCAG 2.1. Primary text pairs (`--fg` on `--bg` and `--fg` on `--surface`) achieve AAA in both themes. Brand color pairs achieve AAA in light and AA in dark.

## Choosing a token

Select the token that matches the semantic role of what you are styling. Never choose a token based on its hex value. Token names and roles are the same across themes — only the assigned value changes with the theme.

## Light or dark mode

The portfolio supports light and dark mode, controlled by the `.dark` class on `<html>`. Every adaptive token resolves to the correct value automatically. You cannot implement light or dark mode without using tokens everywhere — hardcoded hex values will not change when the mode switches.

**Respecting OS preferences**

Most operating systems (macOS, iOS, Windows, Android) support dark and light mode via the `prefers-color-scheme` media query. The `useTheme` hook detects this preference on first load and applies the matching theme automatically. Users can override it using the `ThemeToggle` component.

**What changes automatically**

All eight adaptive tokens update instantly when the theme changes — no re-renders, no component-level conditionals needed. The same component code works correctly in both themes.

**What to watch**

- Images and videos do not respond to theme changes — choose assets that work in both modes, or swap them manually
- Static Tailwind classes like `bg-cream` or `bg-dark-bg` are locked values and will not switch — use them only in Storybook story wrappers, never in component code

## CSS variables

Reference tokens directly as CSS custom properties. Use this in inline styles in JSX or anywhere Tailwind classes are not available.

```jsx
// Inline styles
<div style={{ color: 'var(--fg)', backgroundColor: 'var(--surface)' }}>
  Card content
</div>

// Border using the border token
<div style={{ border: '1px solid var(--border)' }}>…</div>

// Hover state using brand tokens
<button
  style={{ backgroundColor: 'var(--accent)' }}
  onMouseEnter={e => e.currentTarget.style.backgroundColor = 'var(--brand-hover)'}
  onMouseLeave={e => e.currentTarget.style.backgroundColor = 'var(--accent)'}
>
  Button
</button>
```

## Dark mode

Dark mode is controlled by the `.dark` class on `<html>`. The `ThemeToggle` component manages this automatically. All adaptive tokens update instantly — no re-renders needed.

```jsx
// ThemeToggle sets this on click
document.documentElement.classList.toggle('dark')

// useTheme hook — detect OS preference on first load
const prefersDark = window.matchMedia('(prefers-color-scheme: dark)').matches
if (prefersDark) document.documentElement.classList.add('dark')
```
